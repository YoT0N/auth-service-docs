About the System

Auth Service is a microservice responsible for secure user authentication and authorization in a distributed system. It provides a REST API integrated with other microservices via JWT tokens.

System Specification Document

Version: v1.0 Status: Approved Source of truth for: all functional requirements
This document is the Single Source of Truth for functional requirements. SDD, Test Strategy, and Traceability Matrix reference requirement IDs (FR-XX, NFR-XX) defined here.

1. Functional Requirements

FR-01 — User Registration

Accepts user data (email, password), verifies email uniqueness, hashes with bcrypt (cost factor 12), stores record, returns confirmation.

POST/api/auth/register201, 400, 409

FR-02 — Authentication (Login)

Verifies credentials, generates JWT pair (access: 15 min, refresh: 7 days), stores refresh token in Redis.

POST/api/auth/login200, 400, 401

FR-03 — Token Refresh

Accepts refresh token, verifies in Redis, invalidates old token, generates new pair (Refresh Token Rotation).

POST/api/auth/refresh200, 401

FR-04 — Logout

Deletes refresh token from Redis, preventing further session renewal.

POST/api/auth/logout204, 401

FR-05 — Token Verification

Internal endpoint for token verification by other microservices via API Gateway.

GET/api/auth/verify200, 401

Software Design Document

Version: v1.0 References: SSD

1. Architecture

Layered Architecture: Controller → Service → Repository. Stateless design — session state stored externally in Redis.

Client / API Gateway
        │
        ▼
┌─────────────────────┐
│   Controller Layer  │  ← HTTP handling, input validation
├─────────────────────┤
│   Service Layer     │  ← Business logic, token management
├─────────────────────┤
│  Repository Layer   │  ← Data access via Prisma ORM
├─────────────────────┤
│ PostgreSQL │ Redis  │  ← Persistent storage + session cache
└─────────────────────┘

2. Components

ComponentTechnologyResponsibility
API LayerNode.js / ExpressHTTP handling, validation
Business LogicNode.jsAuth logic, token operations
Token Modulejsonwebtoken (RS256)JWT generation and verification
Data LayerPrisma ORMCRUD with PostgreSQL
Cache LayerioredisRefresh token storage
Security MiddlewareHelmet, express-rate-limitAttack protection, rate limiting

3. Login Flow

  1. API Gateway receives POST /api/auth/login from client
  2. Controller validates request body (email, password)
  3. AuthService queries UserRepository for user record
  4. Password verified via bcrypt.compare()
  5. TokenModule generates access_token (RS256, 15 min) and refresh_token
  6. Refresh token stored in Redis with TTL of 7 days
  7. Access token returned in body, refresh token in HttpOnly Cookie

4. Security Design

  • bcrypt cost factor 12 — resistant to brute-force
  • RS256 (asymmetric) — private key signs, public key verifies
  • Refresh Token Rotation — each use invalidates the old token
  • HttpOnly Cookie — JavaScript cannot access refresh token (XSS mitigation)
  • Rate limiting — 5 failed attempts per IP per minute

3. Scaling Principles

  • HPA: auto-scales replicas when CPU > 70%
  • Stateless design: any replica can handle any request
  • Redis Cluster: horizontal cache scaling and fault tolerance
  • PgBouncer: database connection pooling
  • Health checks: /health endpoint for Kubernetes probes

Developer Onboarding Guide

Version: v1.0 References: SDD (architecture), ISD (deployment)

1. Prerequisites

  • Node.js 20.x (LTS)
  • Docker + Docker Compose
  • Git

2. Local Setup

git clone https://github.com/your-org/auth-service.git
cd auth-service
npm install
cp .env.example .env

# Generate JWT keys
mkdir -p keys
openssl genrsa -out keys/private.pem 2048
openssl rsa -in keys/private.pem -pubout -out keys/public.pem

# Start infrastructure (PostgreSQL + Redis)
docker-compose up -d

# Run migrations
npx prisma migrate dev

# Start service
npm run dev
